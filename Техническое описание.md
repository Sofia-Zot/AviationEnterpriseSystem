# Информационная система авиастроительного предприятия — Техническое описание

## 1. Описание проекта

Разрабатывается клиент-серверное приложение для учёта производства на авиастроительном предприятии. Система хранит данные о цехах, участках, бригадах, сотрудниках (рабочие, ИТР, испытатели), видах и экземплярах изделий (самолёты, ракеты, вертолёты и др.), технологических картах, выполнении работ и испытаниях. Приложение должно обеспечивать:

- Авторизацию и разграничение прав доступа (администратор, кадровик, начальник цеха, мастер, бригадир, испытатель, технолог, аналитик).
- Ввод, редактирование и удаление данных через графический интерфейс.
- Выполнение 14 аналитических запросов согласно заданию (перечни изделий, кадровый состав, бригады, работы, испытания и т.д.).

## 2. Архитектура

Приложение построено по **клиент-серверной архитектуре**:

- **Серверная часть** – уже спроектированная и реализованная база данных **PostgreSQL 17**. Все таблицы, ограничения целостности (первичные/внешние ключи, CHECK, UNIQUE), триггеры и хранимые процедуры созданы скриптом `aircraft_plant_full.sql`.
- **Клиентская часть** – монолитное десктопное приложение на **Python** с графическим интерфейсом. Оно подключается к серверу БД через драйвер, отправляет SQL-запросы и отображает результаты.

Взаимодействие компонентов:

1. Пользователь запускает клиент, вводит логин/пароль.
2. Клиент обращается к таблице `users` (или использует роль) в БД для проверки прав.
3. В зависимости от роли приложение показывает разрешённые пункты меню и формы.
4. Все операции (ввод, редактирование, запросы) транслируются в подготовленные SQL-запросы, которые выполняются на сервере.
5. Результаты возвращаются клиенту и отображаются в таблицах, формах или отчётах.

## 4. Схемы баз данных (уже реализованы)

Скрипт `aircraft_plant_full.sql` создаёт следующие таблицы (приведены ключевые):

- `shop` – цеха (`id_shop`, `name`)
- `section` – участки (`id_section`, `name`, `id_shop` → shop)
- `brigade` – бригады (`id_brigade`, `name`, `id_section`, `id_foreman` → worker)
- `employee` – сотрудники (`id_employee`, `last_name`, `first_name`, ...)
- `worker` – рабочие (`id_employee` → employee, `profession`, `rank`, `is_foreman`, `id_brigade`)
- `engineer` – ИТР (`id_employee` → employee, `category`, `position`)
- `tester` – испытатели (`id_employee` → employee, `specialization`, `id_lab` → laboratory)
- `product_category`, `product_type`, `product_instance` – категории, виды и экземпляры изделий
- `aircraft_instance`, `rocket_instance` и др. – подтипы изделий
- `tech_card`, `section_work_list`, `work_step` – технологические карты и этапы работ
- `work_execution` – выполнение работ (факт сборки)
- `test_card`, `test_work_list`, `test_step`, `test_execution` – испытания
- `equipment`, `laboratory`, `shop_lab`, `tester_equipment` – лаборатории, оборудование и связи.

**Процедурные ограничения (триггеры):**

- `trg_check_status_transition` – переход статуса изделия только `in_assembly` → `under_test` → `ready`.
- `trg_check_foreman` – бригадир должен быть рабочим этой бригады с флагом `is_foreman`.
- `trg_auto_to_testing` – автоматический перевод на испытания после завершения всех этапов.
- `trg_unique_foreman` – только один бригадир на бригаду.
- `trg_prevent_tk_delete` – запрет удаления техкарты, если есть незавершённые изделия.

**Хранимые процедуры:**

- `start_assembly(p_serial, p_type, p_shop, ...)` – создаёт экземпляр изделия и первый этап выполнения работы.
- `complete_assembly(p_serial)` – переводит изделие в статус `ready` после проверки завершения всех этапов.

## 6. API-контракты (между клиентом и БД)

Клиент не предоставляет REST API – он напрямую вызывает функции DAO, которые выполняют SQL-запросы. Однако для понимания логики ниже приведены примеры сигнатур методов DAO.

### 6.1. Аутентификация

- `auth_service.authenticate(login, password) -> User`  
  Проверяет `users` таблицу (или использует роли), возвращает объект `User` с полями `login`, `role`, `id_employee`.

### 6.2. CRUD-операции

- `employee_dao.get_all() -> List[Employee]`
- `employee_dao.insert(employee) -> bool`
- `employee_dao.update(employee) -> bool`
- `employee_dao.delete(employee_id) -> bool`

Аналогично для `section`, `brigade`, `product_type`, `product_instance` и т.д.

### 6.3. 14 запросов (QueryDAO)

Каждый запрос реализован как метод, возвращающий список строк или словарей.

**Query 1** – перечень видов изделий по цеху/предприятию  
`query1_product_types(shop_id=None, category_name=None) -> List[str]`

**Query 2** – число и перечень изделий, собранных за период  
`query2_products_count_and_list(shop_id=None, section_id=None, category_name=None, start_date, end_date) -> Tuple[int, List[dict]]`

**Query 3** – кадровый состав (рабочие, ИТР)  
`query3_workers(shop_id=None) -> List[dict]`  
`query3_engineers() -> List[dict]`

**Query 4** – участки и их начальники  
`query4_sections_and_managers(shop_id=None) -> List[dict]`

**Query 5** – перечень работ для изделия  
`query5_work_list_for_product(serial_number) -> List[dict]`

**Query 6** – состав бригад участка/цеха  
`query6_brigade_composition(section_id=None, shop_id=None) -> List[dict]`

**Query 7** – список мастеров (ИТР с должностью 'Master')  
`query7_masters(section_id=None, shop_id=None) -> List[dict]`

**Query 8** – изделия в сборке сейчас  
`query8_products_in_assembly(shop_id=None, section_id=None, category_name=None) -> List[dict]`

**Query 9** – бригады, занятые на сборке изделия  
`query9_brigades_for_product(serial_number) -> List[dict]`

**Query 10** – лаборатории для испытаний изделия  
`query10_labs_for_product(serial_number) -> List[dict]`

**Query 11** – изделия, проходившие испытания в лаборатории за период  
`query11_products_tested_in_lab(lab_id, category_name=None, start_date, end_date) -> List[dict]`

**Query 12** – испытатели, участвовавшие в испытаниях  
`query12_testers_for_product(serial_number, lab_id=None, category_name=None, start_date, end_date) -> List[dict]`

**Query 13** – оборудование, использованное при испытании  
`query13_equipment_for_product(serial_number, lab_id=None, category_name=None, start_date, end_date) -> List[dict]`

**Query 14** – число и перечень изделий в сборке в настоящий момент  
`query14_assembly_current(shop_id=None, section_id=None, category_name=None) -> Tuple[int, List[dict]]`

## 7. Роли и права доступа

В БД созданы роли (администратор, кадровик, начальник цеха, мастер, бригадир, испытатель, технолог, аналитик). В клиенте при логине определяется роль, и на основе этого:

- Администратор – полный доступ ко всем таблицам и запросам.
- Кадровик – редактирование сотрудников, бригад, просмотр запросов 3,4,6,7,9,12.
- Начальник цеха – просмотр и редактирование только своей территории (фильтрация через RLS или в коде).
- Мастер – изменение статуса работ, просмотр.
- Бригадир – просмотр своей бригады, изменение статуса изделий.
- Испытатель – ввод результатов испытаний.
- Технолог – редактирование техкарт.
- Аналитик – только просмотр всех запросов.

В клиенте пункты меню, недоступные для роли, скрываются или блокируются.

## 8. Тестирование

Планируются следующие виды тестов:

- **Unit-тесты** (pytest) для DAO и сервисов (изолированные, с тестовой БД или моками).

Для автоматизации GUI-тестов можно использовать `pytest-qt` (для PyQt5).

-- 3.1 Контроль перехода статуса изделия
CREATE OR REPLACE FUNCTION check_status_transition()
RETURNS TRIGGER AS $$
BEGIN
    IF OLD.status = 'in_assembly' AND NEW.status NOT IN ('under_test', 'in_assembly') THEN
        RAISE EXCEPTION 'Product can only move from in_assembly to under_test';
    END IF;
    IF OLD.status = 'under_test' AND NEW.status NOT IN ('ready', 'under_test') THEN
        RAISE EXCEPTION 'Product can only move from under_test to ready';
    END IF;
    IF OLD.status = 'ready' AND NEW.status != 'ready' THEN
        RAISE EXCEPTION 'Ready product cannot change status';
    END IF;
    RETURN NEW;
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER trg_check_status_transition
    BEFORE UPDATE OF status ON product_instance
    FOR EACH ROW EXECUTE FUNCTION check_status_transition();

-- 3.2 Проверка бригадира
CREATE OR REPLACE FUNCTION check_foreman()
RETURNS TRIGGER AS $$
BEGIN
    IF NEW.id_foreman IS NOT NULL THEN
        IF NOT EXISTS (
            SELECT 1 FROM worker
            WHERE id_employee = NEW.id_foreman
              AND id_brigade = NEW.id_brigade
              AND is_foreman = TRUE
        ) THEN
            RAISE EXCEPTION 'Foreman must be a worker of this brigade and have is_foreman = TRUE';
        END IF;
    END IF;
    RETURN NEW;
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER trg_check_foreman
    BEFORE INSERT OR UPDATE OF id_foreman, id_brigade ON brigade
    FOR EACH ROW EXECUTE FUNCTION check_foreman();

-- 3.3 Автоматический перевод на испытания после завершения всех этапов
CREATE OR REPLACE FUNCTION auto_to_testing()
RETURNS TRIGGER AS $$
DECLARE
    v_status VARCHAR(20);
    all_completed BOOLEAN;
BEGIN
    IF NEW.status = 'completed' THEN
        SELECT status INTO v_status FROM product_instance WHERE serial_number = NEW.serial_number;
        IF v_status = 'in_assembly' THEN
            SELECT NOT EXISTS (
                SELECT 1
                FROM work_step ws
                JOIN tech_card tc ON ws.id_card = tc.id_card
                JOIN product_instance p ON p.id_type = tc.id_type
                LEFT JOIN work_execution we ON we.serial_number = p.serial_number AND we.id_step = ws.id_step
                WHERE p.serial_number = NEW.serial_number
                  AND (we.status IS NULL OR we.status != 'completed')
            ) INTO all_completed;
            IF all_completed THEN
                UPDATE product_instance SET status = 'under_test' WHERE serial_number = NEW.serial_number;
            END IF;
        END IF;
    END IF;
    RETURN NEW;
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER trg_auto_to_testing
    AFTER UPDATE OF status ON work_execution
    FOR EACH ROW EXECUTE FUNCTION auto_to_testing();

-- 3.4 Уникальность бригадира в бригаде
CREATE OR REPLACE FUNCTION unique_foreman_per_brigade()
RETURNS TRIGGER AS $$
BEGIN
    IF NEW.is_foreman = TRUE THEN
        IF EXISTS (
            SELECT 1 FROM worker
            WHERE id_brigade = NEW.id_brigade
              AND is_foreman = TRUE
              AND id_employee != NEW.id_employee
        ) THEN
            RAISE EXCEPTION 'Brigade already has a foreman';
        END IF;
    END IF;
    RETURN NEW;
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER trg_unique_foreman
    BEFORE INSERT OR UPDATE OF is_foreman, id_brigade ON worker
    FOR EACH ROW EXECUTE FUNCTION unique_foreman_per_brigade();

-- 3.5 Запрет удаления техкарты при незавершённых изделиях
CREATE OR REPLACE FUNCTION prevent_tk_delete()
RETURNS TRIGGER AS $$
BEGIN
    IF EXISTS (
        SELECT 1 FROM product_instance
        WHERE id_type = OLD.id_type
          AND status != 'ready'
    ) THEN
        RAISE EXCEPTION 'Cannot delete tech card: unfinished products of this type exist';
    END IF;
    RETURN OLD;
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER trg_prevent_tk_delete
    BEFORE DELETE ON tech_card
    FOR EACH ROW EXECUTE FUNCTION prevent_tk_delete();

-- 4.1 Начало сборки нового изделия
CREATE OR REPLACE PROCEDURE start_assembly(
    p_serial INT,
    p_type INT,
    p_shop INT,
    p_weight NUMERIC(10,2) DEFAULT NULL,
    p_material VARCHAR(100) DEFAULT NULL,
    p_batch VARCHAR(50) DEFAULT NULL
)
LANGUAGE plpgsql AS $$
DECLARE
    first_step INT;
BEGIN
    INSERT INTO product_instance (serial_number, status, id_shop, id_type, weight, material, batch)
    VALUES (p_serial, 'in_assembly', p_shop, p_type, p_weight, p_material, p_batch);

    SELECT ws.id_step INTO first_step
    FROM tech_card tc
    JOIN work_step ws ON tc.id_card = ws.id_card
    WHERE tc.id_type = p_type
    ORDER BY ws.step_number
    LIMIT 1;

    IF first_step IS NULL THEN
        RAISE EXCEPTION 'No tech card defined for product type %', p_type;
    END IF;

    INSERT INTO work_execution (serial_number, id_step, start_date, status)
    VALUES (p_serial, first_step, CURRENT_DATE, 'in_progress');
END;
$$;

-- 4.2 Завершение сборки изделия
CREATE OR REPLACE PROCEDURE complete_assembly(p_serial INT)
LANGUAGE plpgsql AS $$
DECLARE
    all_completed BOOLEAN;
BEGIN
    SELECT NOT EXISTS (
        SELECT 1
        FROM work_step ws
        JOIN tech_card tc ON ws.id_card = tc.id_card
        JOIN product_instance p ON p.id_type = tc.id_type
        LEFT JOIN work_execution we ON we.serial_number = p.serial_number AND we.id_step = ws.id_step
        WHERE p.serial_number = p_serial
          AND (we.status IS NULL OR we.status != 'completed')
    ) INTO all_completed;

    IF NOT all_completed THEN
        RAISE EXCEPTION 'Not all assembly steps are completed for product %', p_serial;
    END IF;

    UPDATE product_instance SET status = 'ready' WHERE serial_number = p_serial;
END;
$$;

## 13. Реализация замечаний преподавателя

### 13.1. Нативные запросы
Все обращения к базе данных выполняются через прямые SQL-запросы без использования объектно-реляционных отображений (ORM). Драйвер psycopg2 обеспечивает нативное взаимодействие с PostgreSQL.

### 13.2. Представления для запросов (VIEW)
Для каждого из 14 обязательных запросов создано представление в базе данных. Это позволяет:
- Упростить код клиента: запросы превращаются в короткие `SELECT * FROM view WHERE ...`
- Централизовать логику на стороне БД
- Применять фильтры (по цеху, участку, категории, дате и т.д.) как дополнительные условия `WHERE`.

Пример для запроса 1 (виды изделий в цехе 1):
```sql
CREATE VIEW product_types_in_shop1 AS
SELECT DISTINCT pt.name AS product_type, pt.id_category
FROM product_type pt
JOIN product_instance pi ON pi.id_type = pt.id_type
WHERE pi.id_shop = 1;
В клиенте:

python
cursor.execute("SELECT product_type FROM product_types_in_shop1 WHERE id_category = %s", (cat_id,))
Аналогичные представления созданы для всех запросов 1–14.

13.3. Использование возможностей PostgreSQL
В скрипте создания БД применяются только функции PostgreSQL: GENERATED BY DEFAULT AS IDENTITY, plpgsql, триггеры, ON CONFLICT, ARRAY_AGG, JSONB (для специфических атрибутов). Стандартный SQL без расширений не используется.

13.4. Управление соединениями (сессиями)
В клиенте реализован пул соединений (psycopg2.pool.SimpleConnectionPool). Каждый вызов DAO работает по схеме:

python
conn = pool.getconn()
try:
    cur = conn.cursor()
    cur.execute(sql, params)
    conn.commit()
finally:
    pool.putconn(conn)
Это гарантирует, что сессия открывается только на время одного запроса и немедленно закрывается. Накопления соединений не происходит.

13.5. Безопасные подстановки параметров
Все запросы используют параметризованные вызовы с плейсхолдерами %s и передачей параметров отдельным аргументом:

python
# Правильно (безопасно)
cursor.execute("SELECT * FROM employee WHERE id = %s", (emp_id,))

# Запрещено (SQL-инъекция)
cursor.execute(f"SELECT * FROM employee WHERE id = {emp_id}")
Термин «combined подстановки» относится именно к такому способу. В коде клиента полностью исключена конкатенация строк при формировании SQL.

13.6. Обработка ошибок без stack trace в диалогах
В слое GUI все исключения перехватываются. Пользователю показывается дружественное сообщение через QMessageBox.critical(). Полный stack trace записывается в лог-файл с помощью модуля logging, но никогда не выводится напрямую в интерфейс.

Пример:

try:
    rows = dao.get_some_data()
except psycopg2.Error as e:
    logger.exception("Database error in query")
    QMessageBox.critical(self, "Ошибка", f"Не удалось выполнить запрос: {e}")
    return
Это соответствует требованию безопасности и удобства пользователя.
